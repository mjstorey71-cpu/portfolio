Building an AI-Native Design System

From fragmenting components to machine-readable infrastructure. How a Head of Product Design transformed the design system into a strategic asset that bridges design intent and AI-powered engineering velocity.

The Challenge

Scaling Design Quality Across Global Teams

By 2025, RentSpree was operating with a growing design system that lacked cohesion. Teams in the US and Thailand were shipping inconsistent components, engineers faced ambiguous handoffs from Figma, and the design system had become a pattern library rather than a strategic asset. Design debt was accumulating faster than it could be paid down.

🔀

Fragmented Components

One-off "snowflake" components created outside the system because consistency requirements weren't clear or components couldn't be extended.

🤝

Handoff Friction

Design-to-code handoffs were error-prone. Engineers struggled to understand token usage, component variants, and design intent.

📐

Governance Vacuum

No formal process for deciding which deviations should become system standards versus accepted exceptions.

🔗

AI Readiness Gap

The system lived in Figma. Code generation tools had no structured way to consume design intent, tokens, or component contracts.

The Role

Head of Product Design and Research

As Head of Product Design and Research at RentSpree, I set the strategic vision for the design system as infrastructure not decoration. I assembled a core team and made critical decisions about prioritization, tooling, and process. I worked cross-functionally with engineering and product leadership to embed the design system into how we build.

The responsibility was three-fold: expand the system's operational coverage, introduce tooling to enforce consistency, and architect it for AI consumption. I led by example, making the design system a production-quality piece of software.

The Vision

From Static Library to Adaptive Experience System

Every design system update in 2025 was made in service of a larger shift: moving RentSpree's design system from a static component library to a behavior-driven experience infrastructure. The system we were building toward doesn't store patterns — it generates them. It learns from use, updates itself, and renders components uniquely per situation.

This direction was a direct response to the structural challenges holding us back: fragmented codebases, inconsistent UI across teams, and no single front-end source of truth. A static library can't solve those problems at scale. An adaptive system can.

The architecture is built around five contextual layers. At the center is the component layer — what we have today. Surrounding it, intent captures what the user is trying to accomplish. Context accounts for where and how they are in the product. Platform adapts to the surface (web, mobile, voice). Personalization shapes the experience around individual behavior. And at the outermost layer, agentic orchestration allows the system to respond autonomously — shipping improvements, reducing user decisions, and making experiences fluid without manual intervention.

A concrete example: a pricing card component that today has fixed variants would instead render differently for a first-time visitor, a returning power user, or someone accessing the product in voice mode while driving. The system reads the context and generates the right output — meeting RentSpree's requirements and standards — without a designer manually creating each variant.

Fixed variants and stale docs give way to behavior-driven UI that learns from use and updates itself. Patterns are generated, not stored. The system responds autonomously, evolves through usage, and reduces user decisions — making components smarter and experiences fluid."

Part 1: Scaling the System

40+ Updates in 2025: Operational Expansion

The system didn't fail us; it was under-resourced. We made a deliberate decision: treat the design system as software, not as an afterthought.

What We Built

Migrated component documentation from bit.dev to Storybook, the industry standard. Added live code playgrounds for every component so engineers could experiment with props and copy working code. Implemented formal contribution guidelines so designers knew how to extend the system safely.

Expanded the token system with typography updates (new type scales with proper letter spacing), updated form patterns to single-column layouts, and added new component variants (Banner system messaging, updated Dialog, enhanced Cards). Each update was documented, tested, and deployed consistently.

Monthly design system recaps became standard communication celebrating wins, highlighting new additions, and clarifying deprecated patterns. This transparency reduced confusion and encouraged adoption.

Key Milestones
  • Jan 2026: Storybook + Live Playgrounds launch
  • Dec 2025: Typography updates & letter-spacing fixes
  • Throughout 2025: Banner, Dialog, Form, Card iterations
  • Monthly: DS Monthly Recaps announced
  • Ongoing: Contribution guidelines enforcement

The goal was clear: we wanted to avoid the pattern where engineering had to guess what the design system was. Every update would be announced, every change would be traceable, and contribution would be guided, not gatekept.

Part 2: DS Auditor Plugin

Quality Gates Before Handoff

Expansion alone wasn't enough. We needed enforcement. The DS Auditor is a custom Figma plugin that scans designs in real time, flagging token misuse and component deviations before designs ever reach engineering.

Why It Matters

Engineering handoffs were a major friction point. Designers would move a design to "ready for dev," but engineers would discover:

  • Hardcoded colors instead of tokens
  • Unsupported component variants
  • Missing accessibility annotations
  • Spacing that didn't align with the token grid

DS Auditor runs as engineers work, catching these issues live. It's a coach, not a blocker suggesting fixes and linking to documentation.

What DS Auditor Checks
Token Alignment
Flags colors, sizes, spacing not in the token system
Component Variants
Ensures only defined variants are used
Accessibility Annotations
Prompts for alt text, ARIA roles where needed
Snowflake Detection
Identifies deviations and suggests escalation

Reducing handoff friction by 45% wasn't just about saving time. It was about shifting trust engineers could now trust that a design marked "ready for dev" had been vetted. Designers got immediate feedback on spec violations without waiting for code review.

Part 3: Snowflake Standardization

Formal Process for Design Variance

A "snowflake" at RentSpree is a component or pattern that doesn't exist in the design system, either because it's new or because it significantly modifies an existing component. We needed a formal way to decide: should this become standard, stay as an exception, or be sunset?

The Process

  1. Designer submits a form when introducing a snowflake, becoming its owner.
  2. Design is reviewed, approved, and launched with clear intent: experimental or intentional exception.
  3. Snowflake is monitored in production usage metrics, engineering feedback, design evolution.
  4. Owner circles back after sufficient data to decide: standardize into the system, keep as snowflake, or sunset.

This isn't bureaucracy it's transparency. It gives designers permission to innovate while keeping the system honest about what's core vs. experimental.

Supporting Infrastructure

Snowflakes aren't a problem untracked snowflakes are. By formalizing the process, we shifted from siloed design chaos to intentional variance with a clear path to resolution.

Part 4: AI-Ready Infrastructure

Figma MCP + Claude Design System Rules

The most forward-looking move: making the design system machine-readable. By February 2026, an engineer on our team discovered the Figma MCP create_design_system_rules command and realized we could codify the entire design language into structured format that AI agents understand.

The Shift

Previously, when using AI to generate code (Claude Code or similar), the agent had context about the component but lacked understanding of RentSpree's design language. It would generate functionally correct code that looked generic.

The Figma MCP + create_design_system_rules workflow changes this. The AI can now:

  • Query the Figma file directly for tokens, styles, and component specs
  • Extract design rules (colors, typography, spacing, border radius patterns)
  • Generate code that reflects RentSpree's specific look and feel
  • Skip guesswork and prototyping AI generates production-quality code aligned to spec

PR #1485 immortalized these rules in our main repo, making it the canonical reference for AI-powered development.

The Workflow
1. Engineer asks Claude: "Generate button component matching our design system"
2. Claude uses Figma MCP to fetch the design file and extract rules
3. Claude references design system rules (tokens, components, patterns)
4. Claude generates code using proper tokens, RentSpree color palette, spacing grid
5. Engineer reviews, approves, and ships 3x faster than manual coding

This isn't about replacing engineers. It's about giving them a design-aware AI partner. The system becomes the source of truth not just for humans, but for machines. Design intent is now executable.

Reflection

What This Means

This case study is about a fundamental shift in how we think about design systems. They're not pattern libraries. They're software infrastructure subject to the same rigor, versioning, and testing discipline as backend APIs.

When you treat the design system as software, you start asking harder questions: How do we version breaking changes? How do we measure adoption? How do we ensure quality? How do we make it accessible to machines, not just humans?

The most surprising outcome: by making the system intentionally rigid (tokens, rules, patterns), we actually made it more flexible. Teams could innovate within guardrails because they understood the rules. Snowflakes became a feature, not a liability.

Key Learnings
Design systems are not design work.
They're software engineering applied to design. Treat them that way.
Enforcement is more important than coverage.
A small system with tooling beats a large system with poor adoption.
Make it machine-readable from the start.
AI is coming. Your design system is more valuable if agents can consume it.
Formal process beats informal chaos.
A clear snowflake standardization process is less bureaucratic than hidden design debt.
